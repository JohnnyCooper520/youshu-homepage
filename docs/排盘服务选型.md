# 有数 · P0 排盘服务选型

> 版本：2026-06-25
> 目标：为大模型报告生成选择一个稳定、可校验、可结构化输出的排盘底座。
> 核心原则：排盘是确定性计算，不交给大模型；大模型只负责基于排盘结果做解释。

## 1. 选型结论

P0 推荐路线：

**先用 `lunar-javascript` 作为排盘基础库，封装一层自己的 `paipan service`，再用命理顾问和第二套工具做交叉校验。**

不建议 P0 直接自研完整八字排盘算法，也不建议一开始依赖外部商业排盘 API。

原因：

- P0 要验证付费报告质量，不应把时间消耗在历法算法自研。
- 排盘结果必须可复现、可测试、可审计。
- 自己封装服务层后，未来可以替换底层库，不影响报告生成链路。

## 2. 候选方案

| 方案 | 适合度 | 优点 | 风险 |
| --- | --- | --- | --- |
| `lunar-javascript` | 高 | MIT；无第三方依赖；覆盖公历、农历、节气、干支、八字、五行、十神等；JS 项目易集成。 | 需要自己封装成产品需要的 JSON；命理流派细节要由顾问校验。 |
| `mystilight-8char` | 中 | 面向八字，声称包含完整八字排盘和分析。 | 生态和使用量较小；需要进一步验证准确性和维护情况。 |
| `@tony801015/chinese-lunar` | 中低 | 可转年柱、月柱、日柱、时柱，简单直接。 | 覆盖年份到 2050；功能偏历法转换，命理结构不够完整。 |
| Python 开源八字库 | 中 | 有些库功能较丰富，适合命理研究和交叉验证。 | 与当前前端/Node 技术栈割裂；工程化和服务部署成本更高。 |
| 第三方商业排盘 API | 中 | 接入快，省开发。 | 成本、稳定性、数据合规、供应商锁定和可解释性风险较高。 |
| 自研排盘算法 | 后期可做 | 完全可控，可沉淀壁垒。 | P0 太重，容易拖慢验证。 |

## 3. 推荐架构

P0 不直接在前端调用排盘库，而是封装一个服务层。

```text
用户表单
  -> paipan service
    -> lunar-javascript
    -> 标准化命盘 JSON
      -> 顾问规则层
        -> 大模型 Prompt
          -> Markdown 报告
```

这样做的好处：

- 前端只提交出生信息和问题，不关心排盘细节。
- 底层库未来可替换。
- 报告 Prompt 只接收稳定 JSON。
- 顾问可以围绕 JSON 字段做规则和质检。

## 5. 顾问校验策略

P0 阶段不要默认库一定正确。至少做三层校验：

### 第一层：固定样本校验

准备 20 个固定样本：

- 不同时辰。
- 不同年份。
- 节气交界日。
- 子时跨日。
- 男女大运顺逆样本。

每个样本由顾问或可靠排盘工具给出标准答案。

### 第二层：双工具交叉

用 `lunar-javascript` 生成结果，再用另一个排盘工具或人工排盘对照。

重点看：

- 年柱是否一致。
- 月柱是否按节气切换。
- 日柱是否一致。
- 时柱是否一致。
- 大运顺逆和起运时间是否一致。

### 第三层：业务字段校验

不是所有命理字段都要给用户看。顾问要确认：

- 哪些字段进入模型。
- 哪些字段只用于内部判断。
- 哪些字段暂时不使用，避免模型过度发挥。

## 6. 技术实施建议

第一步先做本地 Node 脚本，不急着接服务器。

```text
scripts/paipan-demo.js
```

输入：

```bash
node scripts/paipan-demo.js 1996-08-18 12:00 female 深圳
```

输出：

```bash
dist/samples/paipan-19960818.json
```

第二步再封装成服务函数：

```js
calculateBazi(input) -> PaipanResult
```

第三步接入报告生成：

```js
generateAnnualReport(paipanResult, advisorRules) -> markdown
```

## 7. 近期不做的事

P0 不做：

- 紫微斗数、奇门、六爻多体系混合排盘。
- 复杂神煞全量输出。
- 大师流派争议判断。
- 自研完整历法。
- 自动向用户暴露所有术语。

这些都会让 P0 变重，也会让大模型更容易胡说。
